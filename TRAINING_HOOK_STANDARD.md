# Training Hook Standard

## Purpose

Define the minimum training hook surface that every CBMSv21 screen must expose so guided training scenarios can be authored consistently and maintained safely over time.

This standard is for:

- shared training plumbing
- screen-level hook conventions
- scenario authoring expectations
- future screen development

## Shared Platform Hooks

The shared application shell now provides these hooks automatically on routed screens:

- `body[data-cbms-route]`
- `body[data-training-enabled]`
- `body[data-training-scenario-id]`
- `body[data-training-step]`
- `#appMain`
- `#screenContent`

The shared shell also provides stable common targets already used by the fundamentals scenarios:

- `homeNavBtn`
- `appMenuToggleBtn`
- `appMenu`
- `menuJumpInput`
- `menuJumpGoBtn`
- `fyDropdownBtn`
- `verDropdownBtn`
- `dataScopeBtn`
- `dataObjectPickerModal`
- `scopeStatusBtn`
- `langMenu`
- `helpBtn`
- `helpModal`
- `accountNavBtn`
- `logoutNavBtn`

## Shared Runtime Behaviour

When a training scenario is active:

- the training banner is rendered from the shared layout
- the training overlay is rendered from the shared layout
- the current `training_scenario_id` is propagated automatically across internal links and forms
- the active route is exposed as shared screen metadata

This means a screen does not need to manually include the training banner or overlay partials unless there is a special case.

## Automatic Standardized Ids

The shared platform now enforces deterministic ids for rendered screens in two layers:

- server-side in `BaseController` during render
- client-side in the shared layout for dynamic or late-added DOM

This means the runtime DOM is standardized even where older source views still lack manual ids.

This includes:

- forms
- text inputs
- selects
- textareas
- buttons
- action links
- tables
- cards
- alerts
- modals
- Bootstrap tabs

The generated ids are route-based and are intended as a safety net, for example:

- `execution-reservations-search-input`
- `execution-warrants-submit-btn`
- `strategy-setup-save-project-form`
- `strategy-setup-projects-back-btn`

These generated ids are now the platform safety net for the whole application.

For core workflow, training, inquiry, and transaction-entry paths, explicit human-readable ids should still be added in the view markup.

## Screen-Level Hook Rules

Every new or refactored screen should expose stable ids for the key controls a trainee is expected to use.

Minimum recommended targets:

- primary create button
- primary save button
- primary filter button
- reset button
- primary search field
- key workflow action buttons
- major tabs
- any modal that is part of the normal user flow

Naming convention:

- use lowercase kebab case for screen actions
- keep ids stable and descriptive
- prefix by screen family where practical

Examples:

- `projects-create-btn`
- `projects-save-btn`
- `projects-filter-btn`
- `projects-status-tab`
- `commitment-submit-btn`
- `workflow-forward-btn`

Field ids may continue to use existing database-aligned names where already established:

- `ProjectCode`
- `ProjectName`
- `Username`
- `Email`

## Scenario Design Rules

Training scenarios should only target:

- stable shell ids from the shared layout
- stable screen ids explicitly added for the scenario
- controls that are visible and deterministic in the normal flow

Avoid targeting:

- brittle CSS selectors
- text-position based selectors
- transient DOM generated by third-party widgets unless the widget wrapper has a stable id

## Completion Rule Guidance

The generic overlay can reliably support:

- `manual_continue`
- `click_target`
- `field_nonempty`
- `field_email`
- `field_prefilled`
- `checkbox_checked`
- `field_matches_sample`

These rules are screen-agnostic as long as the target id exists.

The following still require screen/controller-specific wiring when scenarios are authored:

- `navigation`
- `submit_success`
- workflow-state dependent completion
- post-save redirect completion
- record-specific route guards

## Development Rule

For all future screens:

1. build the screen to the standard UI pattern
2. add stable control ids during screen construction
3. rely on shared training plumbing from the layout
4. only add controller-specific training logic when the scenario needs save, redirect, or workflow-aware completion

## Practical Meaning

“Training-ready” does not mean every screen already has a full scenario.

It means:

- the shared training layer can appear on the screen
- the active scenario context survives navigation
- scenario authors have a standard hook contract to target

Screen-specific scenario scripting should now be done against this standard rather than ad hoc per screen.
